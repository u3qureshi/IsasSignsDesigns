Isa’s Signs & Designs — Project Handoff

1. Project Goal

Build and operate a custom ecommerce website for a Canadian small business selling:

- Islamic wall art
- Islamic children’s products
- Montessori wooden toys and learning boards
- Personalized nursery decor and lamps
- Ramadan and Eid products
- Nikkah and wedding signs
- Business signs, plaques, QR signs, cards, and custom displays

The project is being developed by one developer and should remain simple to maintain, fast for customers, data-driven, easy to deploy, scalable enough for a small ecommerce business, and usable without changing frontend code whenever products change.

Products are designed and produced in Canada.

6. Header Implementation

The header was heavily iterated and ultimately simplified to avoid flickering.

Final behavior

There are two header levels.

Header 1 contains:

- centered company logo and title logo
- top search input
- optional Made in Canada / Handmade decorative blocks
- branding assets that scroll away naturally

Header 2 contains:

- sticky navigation
- compact logo that appears when Header 2 reaches sticky state
- sticky search icon and expandable search input
- cart icon
- thin bottom border
- promotional shipping bar above the headers

Important decision:

Do not reintroduce animated collapsing header-height logic.

Previous implementations using max-height transitions, fixed/sticky switching, scroll locks, hysteresis, and timed state locking caused repeated flickering and scroll-position glitches.

The working model is intentionally simple:

- Header 1 scrolls away naturally
- Header 2 is always sticky
- compact logo is shown using a simple threshold
- no elaborate transition system

Promotional Bar

A thin dismissible bar exists at the top:

FREE SHIPPING ON ORDERS $100 +  SHOP NOW  [Truck icon]

- sand-themed background
- close button at far right
- SHOP NOW and truck are one clickable link
- link goes to /best-sellers

7. Responsive Header Behavior

Important goals already implemented:

- main logo/title resize using clamp()
- menu spacing and font size resize fluidly
- top search width shrinks gradually
- top search is anchored so its left edge moves while the right edge remains stable
- search must not overlap Made in Canada / Handmade graphics
- side columns collapse rather than reserving invisible width
- Made in Canada and Handmade artwork disappears on smaller widths
- compact sticky logo has a reserved layout slot so navigation does not shift

Avoid large hardcoded width jumps unless necessary.

12. Sale Display

Backend sale field:

{
  "enabled": true,
  "percent": 15
}

Frontend type:

onSale?: {
  enabled: boolean;
  percent: number;
} | null;

For sale products, cards show:

- discounted price in red
- original price in darker gray with strikethrough
- green rounded badge: Save 15%

Discount calculation:

salePrice = priceCents × (100 - percent) / 100

Products without a sale object show the normal price.

17. Backend Class Responsibilities

IsasignsBackendApplication.java

Main Spring Boot entry point.

Responsibilities:

- starts Spring Boot
- triggers component scanning
- initializes Spring MVC
- initializes Spring Data JPA
- initializes Hibernate
- initializes the embedded Tomcat server
- creates application-managed beans

SpringApplication.run(...) starts the dependency injection container and web server.

Product.java

JPA entity representing the products table.

Responsibilities:

- maps Java fields to PostgreSQL columns
- defines database field types and nullability
- allows Hibernate to convert SQL rows into Java Product objects

Important distinction:

Product is the persistence model. It reflects the database structure and should not be used as the public API contract.

ProductRepository.java

Spring Data JPA repository.

Responsibilities:

- query products without manually writing standard SQL
- retrieve active products
- retrieve products by category
- retrieve a product by slug

Category method is case-insensitive, similar to:

findByCategoryIgnoreCaseAndIsActiveTrue(...)

Spring Data parses this method name and generates the SQL behavior automatically.

ProductResponse.java

Public API DTO.

Responsibilities:

- defines exactly what JSON fields API clients receive
- prevents database/JPA internals from leaking into the API
- provides Swagger/OpenAPI field documentation
- converts database-specific structures into frontend-friendly structures

Examples:

- DB images text → API List<String>
- DB tags text[] → API List<String>
- DB on_sale JSONB/raw entity value → API JSON object

ProductController.java

REST API entry point for product requests.

Responsibilities:

- map HTTP routes
- receive query/path parameters
- call ProductRepository
- convert Product entities to ProductResponse DTOs
- parse images and JSON fields
- return HTTP responses
- return 404 when a slug is not found

OpenApiConfig.java

Defines API documentation metadata:

- API title
- description
- version
- local server URL

It affects generated documentation, not product business logic.


18. Product API Endpoints

List all active products:

GET /api/products

Filter active products by category:

GET /api/products?category=Kids

The category filter is case-insensitive:

kids
Kids
KIDS

all return the same Kids products.

Get one product:

GET /api/products/{slug}

Example:

GET /api/products/kids-montessori-puzzle-fish-matching

Missing product:

HTTP 404


19. Backend Data Flow

Example request:

GET /api/products?category=kids

Flow:

1. Browser, Swagger, curl, or frontend sends HTTP request.
2. Embedded Tomcat receives the request.
3. Spring MVC finds ProductController.
4. @RequestParam extracts category=kids.
5. ProductController calls ProductRepository.
6. Spring Data JPA interprets the repository method.
7. Hibernate creates and executes SQL.
8. HikariCP provides a PostgreSQL connection.
9. PostgreSQL queries public.products.
10. PostgreSQL returns matching rows.
11. Hibernate maps each row into a Product entity.
12. Controller converts each Product into ProductResponse.
13. images string is parsed into List<String>.
14. tags String[] is converted to List<String>.
15. onSale data is converted to a JSON object.
16. Jackson serializes ProductResponse objects into JSON.
17. Spring returns HTTP 200 with application/json.
18. Frontend receives JSON and renders product cards.

The products table currently includes:

- id UUID primary key
- slug unique varchar
- name varchar
- description text
- long_description text
- category varchar
- price_cents bigint
- currency varchar
- images text
- material varchar

- created_at timestamptz
- updated_at timestamptz
- is_featured boolean
- stock_qty integer nullable
- is_customizable boolean

Important:

The exact current schema should be verified using:

\d products

There was a discrepancy at one stage where pasted CREATE TABLE output did not show newer columns even though rows contained them. The live DB should be treated as source of truth.


21. JSONB Sale Field

Column:

on_sale jsonb

Example:

{
  "enabled": true,
  "percent": 15
}

Meaning:

- enabled=true means the sale is active
- percent is the percentage markdown
- null means no configured sale

Current backend implementation converts this to a native JSON object in the response.

24. Database Management Rules

Add product:

INSERT a new products row.

Remove product from storefront:

Set is_active = false.

Do not delete unless necessary.

Update product:

Modify the row in PostgreSQL or eventually use an admin page.

Best Sellers:

Use is_featured = true.

Images:

Store Cloudinary public IDs in the images array.

Sales:

Update on_sale JSONB.

Product modifications should not require frontend source-code changes.

27. Important Engineering Decisions

Keep:

- database as product source of truth
- DTOs separate from entities
- Cloudinary public IDs rather than hardcoded delivery URLs
- case-insensitive categories
- simple sticky header
- guest checkout
- is_active soft removal
- price values in cents
- Swagger/OpenAPI documentation
- reusable CSS theme variables

Avoid:

- complex scroll-animation header state
- storing GitHub tokens in remote URLs
- returning JPA entities directly
- putting API secrets in Vite environment variables
- deleting products instead of deactivating
- storing formatted prices such as "$50"
- hardcoding products in frontend components
- using stale local /assets paths for Cloudinary products
- editing the DB without updating init.sql or future migrations
